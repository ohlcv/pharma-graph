### location — 位置层级

用于定位节点在知识体系中的位置，字段顺序固定，只写实际存在的层级：

| 字段 | 来源 | 何时写 |
|---|---|---|
| `book` | 根文件名 | 必须 |
| `part` | 第一级子目录 | 有则写 |
| `chapter` | 第二级子目录或文件名 | 必须 |
| `section` | 第三级子目录 | 有子节文件才写 |
| `subsection` | 第四级子目录 | 有孙节文件才写 |
| `item` | 具体知识点名 | 有则写 |

**重要规则：**
- `section` 和 `subsection` **不能相同**。当章节目录下直接就是内容文件（没有再分 sub-section 子目录）时，`section` 不存在，不要写。
- `section == chapter` 时（如"第二章 生命药学"），删除 `section`，只保留 `chapter`。
- `section == subsection` 时，删除 `subsection`。
- `item` 用于比 `subsection` 更细的知识点（如具体药物名称），有则写。
- `subsection` 只在目录还有第三层子节文件时才出现。

## 字段说明

### data.type — 实体类型

节点"是什么"。同一 type 的节点共享形状，在图中属于同一几何类别。

| 值 | 形状 | 含义 | 示例 |
|---|---|---|---|
| `drug` | 圆角矩形 | 具体药物或制剂 | 美托洛尔、阿司匹林 |
| `disease` | 菱形 | 疾病或病理状态 | 高血压、感冒 |
| `pathogen` | 菱形 | 病原体（细菌/病毒/真菌） | 流感病毒、金黄色葡萄球菌 |
| `mechanism` | 菱形 | 作用机制或信号通路 | β受体阻滞机制 |
| `ingredient` | 六边形 | 辅料、成分、剂型 | 羟丙甲纤维素 |
| `concept` | 椭圆 | 一般概念、术语、章节知识点 | 生物利用度、首过效应 |
| `service` | 六边形 | 药学服务、合理用药 | 用药交代、用药咨询 |
| `pathway` | 矩形 | 信号通路、受体家族 | 肾素-血管紧张素系统 |
| `indicator` | 椭圆 | 检测指标、检验项目 | ALT、血压 |

**注意**：`disease` 和 `pathogen` 虽然形状相同（菱形），但通过 `category` 边框色区分系统领域。

### data.category — 系统归属

节点"属于哪个系统/领域"。同一 category 的节点共享边框色，形成视觉上的系统分区。

| 值 | 颜色（边框） | 含义 |
|---|---|---|
| `cardiovascular` | 红 | 心血管系统 |
| `respiratory` | 蓝 | 呼吸系统 |
| `digestive` | 绿 | 消化系统 |
| `endocrine` | 橙 | 内分泌系统 |
| `musculoskeletal` | 青 | 肌肉骨骼系统 |
| `anti_infective` | 紫 | 抗感染（抗菌/抗病毒/抗真菌） |
| `anti_tumor` | 粉 | 抗肿瘤 |
| `blood` | 玫瑰红 | 血液系统 |
| `immunology` | 金 | 免疫与抗过敏 |
| `dermatology` | 棕 | 皮肤疾病 |
| `antipyretic` | 浅蓝 | 解热镇痛抗炎 |
| `anti_rheumatic` | 深橙 | 抗风湿 |
| `anti_gout` | 紫红 | 抗痛风 |
| `nutrition` | 灰绿 | 营养与维生素 |
| `diagnostic` | 灰 | 诊断与检验 |
| `pharmacy_practice` | 灰蓝 | 药学综合知识与技能 |

**系统归属的作用**：回答"这个药/疾病/概念属于哪一类系统"的问题。
- 两个都是药，但 `category` 不同 → 形状相同、边框色不同（视觉上形成不同色带）
- 同一系统下，一个是药一个是病原体 → 形状不同、边框色相同（视觉上同色区里的不同形状）

### data.layer — 知识层

节点"处在哪一层"。用于布局分组，决定节点在图中的视觉层次（上下左右分区）。

| 值 | 含义 | 在图中的位置倾向 |
|---|---|---|
| `foundation` | 基础层：生理学、病理学、药理学基础知识 | 图的底层/中心 |
| `system` | 药物系统层：具体药物、制剂、成分、辅料 | 图的中层 |
| `clinical` | 临床层：疾病、诊断、治疗方案 | 图的中上层 |
| `service` | 服务层：用药管理、药品调配、合理用药指导 | 图的边缘/外围 |

**layer vs. category 的区别**：
- `category` = "在哪一个系统"（颜色边框）
- `layer` = "在哪一个层次"（布局分区）
- 两者正交：一个药可以在 `system` 层 + `cardiovascular` 系统；一个疾病在 `clinical` 层 + `respiratory` 系统。

### `edges_out` 填写规则

`edges_out` 不是可有可无的装饰字段，而是把正文里“它主要和哪些内容有关”真正落到图上的结构化关系。

#### 什么时候必须写

- 只要这个节点不是整本内容树的绝对根节点，就**默认应该有** `edges_out`。
- 至少要保证：
  - 有一个能说明它“挂在哪”的上级关系；
  - 再根据正文补 1–3 个最关键的关联关系。
- 如果一个节点正文已经明确提到“它属于……”“它和……一起理解”“它会影响……”但 `edges_out` 为空，就不能算完成。

#### 从正文怎么反推 `edges_out`

优先看正文第三问“它主要和哪些内容有关”，再结合第五问“它放在哪里”，按下面顺序落边：

1. **先补结构归属边**
   - 入口类节点、章节点、节节点、具体知识点，通常都应先写一个 `part_of`。
   - 例：节 → 章，章 → 篇，知识点 → 所属节。

2. **再补最关键的关联边**
   - 正文里出现“和……一起理解”“会影响……”“承接……走向……”时，通常补 `relates`。
   - 只写最能帮助用户搭主线的 1–3 条，不要把提到过的所有名词都堆成边。

3. **最后判断是否需要更强语义**
   - `part_of`：属于某个更大的结构；
   - `isa`：某个具体药物/实例属于某个类别；
   - `treats`：药物治疗某疾病；
   - `mechanism`：药物/疾病与某机制直接相关；
   - `relates`：一般相关，但又不适合更强语义时使用。

#### 节点类型的最低建议

- **篇 / 章 / 节入口类**：至少 1 条 `part_of`，再补 1–3 条下游或同主线 `relates`
- **概念类**：至少 1 条 `part_of` 或所属结构边，再补最关键的上下游概念边
- **具体药物类**：至少 1 条所属结构边，通常还应补同剂型 / 机制 / 适应证相关边
- **制剂 / 工艺类**：至少 1 条所属结构边，通常还应补吸收、生物利用度、具体药物实例等关联边

#### 编写时的限制

- 不要为了“有边而有边”，写正文里没有体现的牵强关系。
- 不要一次写太多弱关系；少而准优于多而乱。
- `target` 必须指向仓库里真实存在的节点 `id`，不能凭空发明。
- 写完后应运行校验；如果 `edges_out.target` 找不到对应节点，这一条边不能算完成。

#### 完成标准补充

一个节点要算“已完成”，至少同时满足：

- frontmatter 符合当前 schema；
- 正文符合当前五问规范；
- `edges_out` 不为空，且能反映正文里的关键关系；
- `edges_out.target` 全部能在现有节点 id 中找到。

#### 推荐写作顺序：先立主链，再扩分支

这段是当前这轮批量整理后总结出来的**推荐顺序**。它不是唯一可行方法，但对这个项目目前的内容结构最稳、返工最少。

##### 一、推荐顺序是什么

写一个新模块或回补一批旧节点时，优先按下面顺序推进：

1. **先做一个样板链条**
   - 不要一上来全库铺开。
   - 先挑一条比较完整、层级清楚、概念关联明显的小链条做样板。
   - 最好同时包含：篇级入口 / 章节入口 / 节点 / 具体概念或实例。
   - 目的不是“先做最容易的”，而是先把这一类节点的写法、边的密度、正文口径统一出来。

2. **再补这条链的直接上游入口**
   - 当一个样板链条写顺以后，不要立刻跳到很远的别处。
   - 应先回头补它的直接上级：节的上级章、章的上级篇、篇的上级总入口。
   - 这样能尽快形成一条从总入口到具体知识点的可导航主线。

3. **再补同一链上的直接下游兄弟节点**
   - 上游打通以后，再补同章、同节、同主题下最直接的兄弟节点。
   - 比如已经写了 `药物吸收`，就继续补 `生物利用度`、`首过效应`，而不是突然跳去另一本书。
   - 这一阶段的目标，是让同一条学习主线内部先变得连续。

4. **再扩到相邻主链的篇级入口和章节入口**
   - 当一条主链已经有了“入口 → 章节 → 节点”的完整路径后，再去补相邻模块。
   - 优先补篇级入口和最近一级章节入口，因为这些节点承担导航作用，性价比最高。
   - 目前实践下来，`药剂学 → 药动学 → 生命药学 → 药物化学` 这种扩法就比较顺。

5. **最后回头复查“能过校验但关系偏薄”的节点**
   - 有些节点 schema 已经合格，但 `edges_out` 很空，正文第三问也没有真正落边。
   - 这类节点不能算真的完成，应该在主链铺出来以后回头补。
   - 重点看三类：总入口、篇/章入口、正文提到很多关联但边只有 0–1 条的节点。

##### 二、为什么推荐这个顺序

这样做有几个非常实际的好处：

- **先定样板，能统一口径**
  - 如果没有样板，后面每写一批节点都可能换一种正文风格、换一种边密度，最后很难收敛。
  - 先做一条完整链，相当于先定“这一类节点到底算写到什么程度才叫完成”。

- **先打主链，图谱会更早可用**
  - 用户真正需要的不是一堆孤立完成的点，而是能沿着主线一路点下去。
  - 先补上游入口和最近一级章节入口，能最快把图谱从“能看点”变成“能导航”。

- **减少返工**
  - 如果一开始就四处散写，后面一旦正文规范、边密度标准、入口写法调整，返工会成片发生。
  - 先在一条链上把标准试稳，再横向扩散，返工明显更少。

- **更容易发现真正缺的节点和关系**
  - 当一条链顺着写下来时，很容易看到哪里断了：是缺上级入口，缺兄弟节点，还是缺跨概念关系。
  - 如果只孤立写单文件，这种“哪里断链”不容易看出来。

- **校验结果更有意义**
  - 只有当主链先比较完整时，`validate` 的通过才更接近“结构真的通了”。
  - 否则很多节点虽然 technically 过校验，但用户点进去仍然走不动。

##### 三、我这轮实践里的具体心得

结合这次实际整理，当前最有效的方法大致是：

- **先用一条链跑通标准**
  - 例如先用 `药动学 → 药物的体内过程 → 药物吸收/分布代谢排泄` 跑通写法。

- **再用这条链反推同类节点的完成标准**
  - 看看篇级入口应该有多少边、章节入口正文第三问该写到什么密度、概念节点至少要落哪些关系。

- **优先补“入口价值高”的节点**
  - 同样花一次改动，篇级入口和章节入口带来的导航收益通常高于零散小点。

- **把“通过校验”和“真正完成”分开看**
  - 通过校验只代表字段和引用没坏。
  - 真正完成还要求：正文五问成立、第三问能落到 `edges_out`、用户点开后能沿主线继续走。

- **每轮都要小闭环**
  - 每补完一小批节点，就立刻运行校验。
  - 这样可以保证扩写速度和结构可靠性同时存在，不会越写越乱。

##### 四、什么时候可以不按这个顺序

以下情况可以局部调整：

- 某个节点虽然不是入口，但它是很多边共同指向的关键枢纽节点，那可以优先补它。
- 某个上游入口已经写得足够好，而下游某一簇明显断链，可以先修那一簇。
- 用户明确指定只修某一批文件时，以用户要求为准，不强行回头补整条链。

##### 五、一句话原则

**先用样板定标准，再用主链建导航，最后回补通过校验但仍然“走不通”的薄弱节点。**

#### tags — 标签

用**中文**填写，反映知识点的关键词/类别，用于检索和筛选。

**示例**：`tags: [药剂学, 口服制剂, 生物利用度, 药代动力学]`

### 正文格式规范

正文用于在详情面板中展示给用户。正文的核心目标不是简单解释当前节点，而是**帮助用户一边理解单个节点，一边理解节点和节点之间的关系，最后逐步搭出自己的知识体系**。

换句话说，这个项目的目标不是“把每个点分别讲明白”就结束了，而是要让用户在漫游过程中，慢慢看懂：

- 这个节点本身是什么；
- 它和哪些内容有关；
- 它在前后知识链条里起什么作用；
- 它在整本书和整张图谱里放在哪里；
- 它和其他节点连起来以后，整体结构是什么样。

因此，正文必须服务于“**从点到线，再到面**”的学习过程：

- **点**：先搞懂当前节点。
- **线**：再搞懂它和上下游节点、同类节点、相关节点的关系。
- **面**：最后把这些关系收束成自己的整体知识结构。

#### 为什么采用“五问学习法”

五问的意义，不是做一个固定格式的问答模板，更不是机械地把 `summary` 换个说法抄一遍。

五问真正承担的任务是：

- 把 frontmatter 里的专业术语、字段分类、层次信息、关系类型，**翻译成大白话问题**；
- 让用户不是在“读参数”，而是在“回答自己真正关心的问题”；
- 通过连续五个角度，把一个节点从“孤立信息”变成“可放进脑中地图的位置”；
- 帮助用户在反复漫游中形成属于自己的知识体系，而不是只背几个标签。

所以，五问一定要满足两个要求：

1. **问题必须是人话**。
   - 用户面对的不是数据库字段，而是“这到底是什么”“它有什么用”“它和什么有关系”这种能直接进入思考的问题。

2. **回答必须是人话**。
   - 回答不能停留在枚举值复述上，而要把 `type`、`category`、`layer`、`location`、`tags`、`edges_out` 这些结构化信息真正讲活。

#### 五问不是死句子，而是五个维度

正文仍然建议保持“5 问 + 5 答”，但要理解：

- **固定的是五个维度，不是唯一一句问法。**
- 不同类型的节点，可以在同一维度下换更自然的问法。
- 只要最后覆盖了这五个维度，就算合格。

这五个维度分别是：

1. **本体维度**：这个节点到底是什么。
2. **意义维度**：理解它有什么用，为什么值得学。
3. **关联维度**：它主要和哪些内容、对象、机制、上下游主题有关。
4. **过程维度**：它通常出现在什么阶段、流程、场景或学习顺序里。
5. **结构维度**：它在整本书、整张图谱、整套知识框架里放在哪里。

#### 五个维度的所有常用问法

下面不是要求每个节点全部使用，而是提供一个“问法池”。写正文时，要根据节点类型，给每个维度选最自然的一句。

##### 1. 本体维度：它到底是什么

可选问法：

- **它是什么？**
- **这讲的到底是什么？**
- **这个节点说的是哪类东西？**
- **它本质上是在讲什么？**
- **这是一个什么类型的内容？**
- **这是一个具体药物，还是一个概念/机制/章节入口？**
- **这不是在讲哪个点，而是在讲哪一整块内容？**（适合篇/章/节入口）

适用目的：

- 把 `label + type + summary.short` 翻译成人话。
- 先让用户知道自己眼前看到的到底是药、概念、机制、疾病、制剂、章节总纲还是模块入口。

回答要求：

- 第一眼先定身份，不要一上来展开细节。
- 优先回答“它属于哪一类”。
- 如果是入口类节点，要明确说“这是某一篇/某一章的总入口”，不要硬说成普通知识点。
- 如果是具体药物，要直接说“这是一个具体药物”。

##### 2. 意义维度：为什么要理解它

可选问法：

- **它有什么用，为什么要理解它？**
- **学这个有什么意义？**
- **为什么这个节点值得单独记住？**
- **搞懂它能帮助我解决什么问题？**
- **这个内容的重要性体现在哪？**
- **如果这个点没搞懂，后面会卡在哪？**
- **它对后续学习或临床理解有什么帮助？**

适用目的：

- 把 `summary.full + layer` 变成学习价值、应用价值、承上启下价值。
- 让用户知道这个节点不是“看过就完”，而是在整体学习路径里有功能。

回答要求：

- 不要只说“很重要”。要说清“重要在哪里”。
- 可以从三种价值里选一种或几种来写：
  - **学习价值**：是后续知识的前提；
  - **理解价值**：能解释某类现象或规律；
  - **应用价值**：能用于给药、判断、分析、临床理解。
- 如果节点偏基础，就强调它是“底层框架/前提”；如果节点偏应用，就强调它“能拿来分析什么”。

##### 3. 关联维度：它主要和什么有关

可选问法：

- **它主要和什么内容有关？**
- **它通常会和哪些东西一起出现？**
- **理解它时，要顺带想到哪些相关内容？**
- **它和哪些节点最容易连起来理解？**
- **它背后连着哪些主题、对象或机制？**
- **它更适合放在哪个系统或哪条知识链里理解？**
- **从图谱关系看，它最常和谁发生联系？**

适用目的：

- 把 `category + tags + edges_out` 变成人能直接联想的关联网络。
- 这一问的重点不是“谁”，而是“它和哪些内容一起构成意义”。

回答要求：

- 不要只堆标签，要把标签组织成有逻辑的一组关系。
- 可以从以下几类关系里挑重点写：
  - 同系统内容；
  - 上下游过程；
  - 作用对象/服务对象；
  - 相关机制；
  - 所属模块；
  - 直接连接的其他节点类型。
- 如果 `edges_out` 明确，就把边的含义翻译出来；不要直接说 `part_of`、`isa`、`mechanism`，而要说“它属于……”“它通过……发挥作用”“它和……一起理解更顺”。

##### 4. 过程维度：它在什么阶段出现

可选问法：

- **它通常在什么阶段出现或被使用？**
- **它一般是在什么过程中出现的？**
- **它更适合放在什么学习顺序里理解？**
- **它通常发生在前面、中间，还是后面？**
- **它在实际应用里常出现在什么环节？**
- **它是基础起点，过程环节，还是结果判断？**
- **如果按流程看，它通常卡在哪一步？**

适用目的：

- 把 `layer` 和节点本身的流程位置、学习顺序、使用时机结合起来。
- 这一问解决“先后顺序”问题，不解决“书里放哪”问题。

回答要求：

- 明确这是“过程/阶段/顺序”，不要写成位置介绍。
- 可以从以下角度选一个最适合的：
  - 学习顺序：先学、过渡、后续深化；
  - 体内过程：吸收前、吸收后、分布代谢排泄阶段；
  - 临床流程：诊断前提、用药选择、疗效判断、不良反应监测；
  - 机制链条：起点、传导过程、结果表现。
- 目的就是让用户脑中出现“它大概在第几步”的感觉。

##### 5. 结构维度：它放在哪里

可选问法：

- **它在整套书和知识体系里放在哪里？**
- **它在这本书里属于哪一层？**
- **它挂在什么篇、章、节下面？**
- **它在整张知识图谱里处在什么位置？**
- **它属于哪条主线，归在哪个模块下面？**
- **如果从整体框架看，它应该放在哪一块？**
- **它和上一级、下一级结构是什么关系？**

适用目的：

- 把 `location` 和总体结构位置翻译出来。
- 这一问重点回答“归属”和“层级”，帮助用户定位。

回答要求：

- 至少交代清楚它属于哪本书、哪一篇/章/节或哪条主线。
- 最好同时说清它是“总入口”“中间层节点”还是“更具体的下位节点”。
- 可以把书本层级和知识结构层级一起说，但不要重复过程维度里的时间顺序。

#### 不同节点类型，怎么组织不同的五问

五问的维度固定，但不同类型的节点，问法要顺着节点本身来。

##### 1. 具体药物类节点

常见问法倾向：

- 它是什么药？
- 它为什么值得记？
- 它主要和什么机制、疾病、系统或同类药有关？
- 它一般在什么用药或学习阶段出现？
- 它在整套药学知识里挂在哪个系统/章节下？

回答重点：

- 本体：先说是具体药物；
- 意义：强调临床用途、代表性、理解同类药的入口价值；
- 关联：常连机制、适应证、不良反应、同类药；
- 过程：常连给药、起效、代谢、临床使用环节；
- 结构：常挂在某系统药物或某章具体药物部分下面。

##### 2. 概念类节点

常见问法倾向：

- 这个概念到底在讲什么？
- 为什么这个概念是基础？
- 它和哪些现象、规律、判断直接相关？
- 它通常在什么学习阶段先建立？
- 它在整套框架里是底层概念、过渡概念还是应用概念？

回答重点：

- 本体：先界定概念边界；
- 意义：强调它解释什么规律；
- 关联：常连标签、同章概念、后续判断；
- 过程：常连学习顺序；
- 结构：常是某一章或某一系统中的基础支点。

##### 3. 机制类节点

常见问法倾向：

- 这个机制讲的到底是哪一步怎么起作用？
- 理解这个机制能帮我看懂什么？
- 它主要和哪些药物、受体、通路、结果有关？
- 它通常发生在整个过程的哪个环节？
- 它在整章或整条作用链里放在哪？

回答重点：

- 本体：先说这是作用机制/过程机制；
- 意义：强调它能解释药效、不良反应、差异；
- 关联：常连药物、受体、靶点、通路、结果；
- 过程：特别适合写前后步骤；
- 结构：放进“哪条作用链/哪章机制部分”里说明。

##### 4. 疾病 / 状态 / 结果类节点

常见问法倾向：

- 这讲的是哪种疾病、状态或结果？
- 为什么理解它很关键？
- 它和哪些病因、表现、用药或机制有关？
- 它一般出现在疾病发展的哪个阶段或临床判断的哪个环节？
- 它在整套知识里挂在哪个系统或专题下？

回答重点：

- 本体：先说它是疾病、状态还是结果；
- 意义：强调它是治疗对象、判断依据或临床后果；
- 关联：常连病因、表现、治疗、机制；
- 过程：可写疾病发展顺序或临床决策环节；
- 结构：按系统或专题定位。

##### 5. 制剂 / 给药途径 / 工艺类节点

常见问法倾向：

- 这是什么制剂/途径/工艺？
- 为什么要学它，它解决什么实际问题？
- 它和哪些剂型特点、吸收规律、使用场景有关？
- 它通常在给药流程或制剂设计的哪个环节出现？
- 它在药剂学框架里放在哪一块？

回答重点：

- 本体：明确它是剂型、途径还是工艺；
- 意义：强调改善吸收、提高稳定性、方便使用等价值；
- 关联：常连辅料、吸收、临床应用、工艺特点；
- 过程：常连制备或给药流程；
- 结构：常挂在药剂学的某篇某章下面。

##### 6. 篇 / 章 / 节入口类节点

常见问法倾向：

- 这一篇/章/节主要在讲什么？
- 为什么这一块是整个模块里的关键入口？
- 它下面通常会展开哪些子主题？
- 它通常出现在整个学习路径的什么阶段？
- 它在整本书的大框架里处在哪个位置？

回答重点：

- 本体：直接说这是模块入口、章节总纲、内容总入口；
- 意义：强调承上启下和统摄作用；
- 关联：常连下级主题和所属模块；
- 过程：常写学习顺序中的位置；
- 结构：特别强调上下级层级关系。

#### 节点类型 → 推荐五问组合速查表

这一节的目标不是再讲一遍原理，而是给后续批量写节点时一个可以直接套用的“起手模板”。

使用方法：

- 先判断当前节点属于哪一类；
- 再从下面这一类里直接选一组最顺口的五问；
- 如果节点有特殊性，可以微调字面问法，但不要丢掉五个维度。

##### 1. 具体药物类

推荐五问：

1. **它是什么药？**
2. **它为什么值得单独记住？**
3. **它主要和什么机制、疾病或同类药有关？**
4. **它一般在什么用药或学习阶段出现？**
5. **它在整套药学知识里挂在哪个系统或章节下？**

适用场景：

- `type: drug`
- 有明确适应证、机制、同类归属、临床用途的节点

回答提醒：

- 第一问先定性为“具体药物”；
- 第三问优先把机制、治疗对象、同类药串起来；
- 第四问尽量写出它在给药、起效、代谢或临床使用链条里的位置。

##### 2. 概念类

推荐五问：

1. **这个概念到底在讲什么？**
2. **为什么这个概念要先搞懂？**
3. **它主要和哪些规律、现象或判断有关？**
4. **它通常在什么学习阶段先建立？**
5. **它在整套框架里属于哪一层、放在哪一块？**

适用场景：

- `type: concept`
- 药理、药动、药剂、生命药学里的基础概念节点

回答提醒：

- 不要只下定义，要把“它解释什么”说出来；
- 第三问常用来连接同章概念和后续判断；
- 第四问通常写成学习顺序，而不是临床流程。

##### 3. 机制类

推荐五问：

1. **这个机制讲的到底是哪一步怎么起作用？**
2. **理解这个机制能帮我看懂什么？**
3. **它主要和哪些药物、受体、通路或结果有关？**
4. **它通常发生在整个过程的哪个环节？**
5. **它在整条作用链或整章内容里放在哪？**

适用场景：

- `type: mechanism`
- 与受体、靶点、通路、作用过程直接相关的节点

回答提醒：

- 第一问要把“怎么起作用”讲出来；
- 第三问优先连接药物、靶点、结果；
- 第四问适合明确前后步骤，不要写空泛时间词。

##### 4. 疾病 / 状态 / 结果类

推荐五问：

1. **这讲的是哪种疾病、状态或结果？**
2. **为什么理解它很关键？**
3. **它和哪些病因、表现、用药或机制有关？**
4. **它一般出现在疾病发展的哪个阶段或临床判断的哪个环节？**
5. **它在整套知识里挂在哪个系统或专题下？**

适用场景：

- 疾病、病理状态、临床结果、不良结局等节点

回答提醒：

- 第一问先分清它是病、状态还是结果；
- 第三问常把病因、表现、治疗对象连成一组；
- 第四问可以写疾病演变顺序或临床决策流程。

##### 5. 制剂 / 给药途径 / 工艺类

推荐五问：

1. **这是什么制剂、给药途径或工艺？**
2. **为什么要学它，它解决什么实际问题？**
3. **它主要和哪些剂型特点、吸收规律或使用场景有关？**
4. **它通常在给药流程或制剂设计的哪个环节出现？**
5. **它在药剂学框架里放在哪一块？**

适用场景：

- 剂型、给药途径、制备工艺、辅料相关节点

回答提醒：

- 第二问尽量写出“解决什么问题”；
- 第三问多连接吸收、稳定性、使用便利性；
- 第四问可写制备流程或给药流程，不要只写“很常见”。

##### 6. 篇 / 章 / 节入口类

推荐五问：

1. **这一篇、这一章或这一节主要在讲什么？**
2. **为什么它是这一块内容的关键入口？**
3. **它下面通常会展开哪些子主题？**
4. **它通常出现在整个学习路径的什么阶段？**
5. **它在整本书的大框架里处在哪个位置？**

适用场景：

- 篇级入口、章级总纲、节级总入口等结构节点

回答提醒：

- 第一问直接说“这是总入口/总纲”；
- 第三问要体现它往下能展开什么，不要只重复标题；
- 第五问特别适合把上下级层级一起说清楚。

##### 7. 不确定类型时的保底组合

如果一时拿不准节点更像哪一类，可以先用这组最稳妥的保底五问：

1. **它是什么？**
2. **为什么要理解它？**
3. **它主要和什么内容有关？**
4. **它通常在什么阶段出现或被使用？**
5. **它在整套书和知识体系里放在哪里？**

这组问法最通用，适合先写出一版，再根据节点特征做精细化调整。

#### 怎么从 frontmatter 判断节点属于哪一类

为了后续批量生成正文稳定一致，不能只靠直觉判断节点类型，而要优先根据 frontmatter 做识别。

判断顺序建议如下：

##### 第一步：先看 `data.type`

`data.type` 是第一判断依据，优先级最高。

常见对应关系：

- `type: drug` → 优先按“具体药物类”处理
- `type: concept` → 优先按“概念类”处理
- `type: mechanism` → 优先按“机制类”处理
- 如果后续扩展出疾病、制剂、路径、章节入口等更细类型，也优先按类型直归

基本原则：

- **能靠 `type` 直接判断，就不要绕远路。**
- `type` 决定这是什么东西，是选哪套五问的第一信号。

##### 第二步：再看 `location` 判断它是不是结构入口

即使 `type` 没有单独写成“chapter / section / module”，也可以通过 `location` 和文件位置判断它是不是入口类节点。

常见信号：

- 文件名与当前篇、章、节标题一致；
- `location` 只到篇、章、节这一层，明显承担结构总入口功能；
- `summary` 描述的是“本章介绍……”“本节围绕……展开……”“这一篇主要讲……”这类总述句；
- `edges_out` 更偏向连下级主题或所属模块，而不是描述具体药物/机制细节。

满足这些信号时，应优先按“篇 / 章 / 节入口类”来组织五问。

也就是说：

- **入口类的判断，有时比 `type: concept` 更重要。**
- 有些结构节点虽然写成 `concept`，但实际写作时不能当普通概念写，而要当“总入口/总纲”写。

##### 第三步：再看 `summary` 判断它更像“定义”还是“总述”

`summary.short` 和 `summary.full` 可以帮助区分它更像单点概念，还是一整块内容入口。

判断方法：

- 如果 `summary` 更像下定义、讲规律、讲某种机制，通常更接近概念类或机制类；
- 如果 `summary` 更像总览、导读、统摄多个子主题，通常更接近篇章入口类；
- 如果 `summary` 明显强调疾病表现、治疗对象、临床结果，就更接近疾病 / 状态 / 结果类；
- 如果 `summary` 明显强调剂型特点、给药方式、工艺目标，就更接近制剂 / 给药途径 / 工艺类。

##### 第四步：看 `tags + edges_out` 判断它更靠哪条知识链

当 `type` 不够细，或者一个节点既像概念又像过程时，可以用 `tags` 和 `edges_out` 再判断一次。

常见判断方式：

- 标签和边大多连药物、适应证、同类药 → 更偏药物类
- 标签和边大多连受体、通路、靶点、作用结果 → 更偏机制类
- 标签和边大多连病因、表现、治疗、结局 → 更偏疾病 / 状态 / 结果类
- 标签和边大多连剂型、吸收、辅料、给药方式 → 更偏制剂 / 给药途径 / 工艺类
- 标签和边大多连下级主题、所属篇章、模块归属 → 更偏篇章入口类

##### 第五步：如果仍然模糊，就先用保底组合

如果一个节点跨得比较宽，或者当前 frontmatter 信息还不足以明确细分类型，就不要硬判。

这时做法是：

- 问句先用“保底五问”；
- 回答里再通过语气把它写得更像概念、入口、机制或过程；
- 等后续 schema 更细时，再回头细分。

核心原则不是“标签必须判得绝对准”，而是“**五问最终要贴合用户理解这个节点的最佳方式**”。

#### 生成正文时的执行流程

为了保证不同节点写出来的正文风格统一、信息完整、又不死板，建议按下面流程生成。

##### 第 1 步：先识别节点类型

按前面的判断规则，先判断它更像：

- 具体药物
- 概念
- 机制
- 疾病 / 状态 / 结果
- 制剂 / 给药途径 / 工艺
- 篇 / 章 / 节入口
- 或暂时不确定

这一步决定后面优先选哪组五问。

##### 第 2 步：从五个维度各选一个最自然的问题

不是把模板整段照抄，而是：

- 本体维度选 1 问
- 意义维度选 1 问
- 关联维度选 1 问
- 过程维度选 1 问
- 结构维度选 1 问

要求：

- 问句要像人会真的问出来的话；
- 五问合起来要顺；
- 问句字面可以变，但五个维度不能缺。

##### 第 3 步：把 frontmatter 拆成“可说的人话信息”

写回答前，不要急着直接输出，而要先在脑中把字段转义一遍：

- `type` → 这是药物、概念、机制，还是入口
- `category` → 它更适合放在哪个系统/模块里理解
- `layer` → 它更偏基础、过渡，还是应用
- `location` → 它在整本书和图谱里挂在哪
- `tags` → 它通常和哪些主题一起出现
- `edges_out` → 它和哪些节点存在“属于 / 机制 / 治疗 / 导致 / 相关”等关系

这一步的关键是：**先理解字段，再写句子**，不要把字段值直接搬进正文。

##### 第 4 步：五答分别解决五个不同问题

生成回答时，要确保每一答都承担不同功能：

- 第一答定身份
- 第二答讲价值
- 第三答讲关联
- 第四答讲顺序/阶段
- 第五答讲结构位置

自检标准：

- 去掉任意一答，信息都会少一块；
- 任意两答之间不要大面积重复；
- 尤其“阶段”和“位置”不能混成一件事。

##### 第 5 步：把关系讲出来，不要只讲定义

正文好不好，关键看第三答和第五答有没有真正把网络感写出来。

要主动做到：

- 让用户知道它和哪些节点/主题一起理解；
- 让用户知道它在前后知识链里接哪一段；
- 让用户知道它在整本书和整张图谱里摆在哪。

如果五答写完后，只剩下“定义 + 重要性 + 路径复述”，那说明关系感还不够。

##### 第 6 步：最后检查是不是“大白话”

最终输出前要做一次语言检查：

- 有没有直接复述英文枚举值；
- 有没有太像教材定义原文；
- 有没有只是在罗列 tags；
- 有没有把 `edges_out` 说成技术字段，而不是自然关系；
- 有没有让普通人读完以后，真的觉得“我知道它是什么、跟谁有关、放在哪了”。

只要还有“在报字段”的感觉，就说明还没真正完成翻译。

#### 选五问时的原则

写具体节点时，不要求机械照抄同一套问句，而要按以下原则选问法：

1. **维度不能少，但问句可以换。**
   - 五个维度都要覆盖。
   - 但“它是什么？”可以换成“这讲的到底是什么？”；“它主要和什么内容有关？”也可以换成“理解它时，要顺带想到哪些相关内容？”。

2. **问句要和节点类型贴脸。**
   - 药物节点就别问得太抽象；
   - 章节节点就别问得像具体药物说明书；
   - 机制节点要把“怎么起作用”感带出来。

3. **问句要服务于关系理解。**
   - 不是只讲单点定义，而是要把用户往“它和谁连着、为什么连、放在哪条线里”上带。

4. **一问只解决一个维度，不要串台。**
   - “阶段”不要写成“位置”；
   - “位置”不要写成“作用”；
   - “关联”不要退化成简单标签罗列。

#### 怎么回答，才算真正把 frontmatter 吃透

写回答时，要把 frontmatter 当成“素材层”，而不是“最终呈现层”。

也就是说：

- `type` 不是原样抄出来，而是翻译成“具体药物 / 基础概念 / 机制 / 模块入口 / 章节总纲”等人话。
- `category` 不是只说分类名，而是说“更适合放在……系统/模块里理解”。
- `layer` 不是只说“基础层/应用层”，而是说“这是前面必须先建立的底层认识”或“已经更靠近临床使用”。
- `location` 不是只抄路径，而是说“它位于哪一篇哪一章，在整体结构里是什么位置”。
- `tags` 不是简单逗号拼接，而是组织成“它主要和……这些内容一起理解”。
- `edges_out` 不是报边类型，而是翻成“它属于……”“它通过……起作用”“它会引出……”“它常和……连起来看”。

#### 回答的写法要求

- 每一问都要显式写标题。
- 每一答都用大白话，避免英文枚举值直接出现。
- 每一答以 1–3 句为宜。
- 五答合起来，通常控制在约 180–320 字；复杂节点可以略长，但不要写成大段散文。
- 不要为了凑五问而写空话。
- 真正好的回答，读完以后用户脑中应该同时留下三样东西：
  - 当前节点是什么；
  - 它和哪些节点/内容连着；
  - 它在整体结构里摆在哪。

#### 正文模板示例

```markdown
#### 它是什么？
它是药理学里的一个基础概念，讲的是药物作用并不只有我们想要的治疗效果，还可能同时伴随不良反应。

#### 它有什么用，为什么要理解它？
理解这个概念，可以帮助判断用药时为什么总要权衡疗效和风险，也能解释为什么同一种药既能治病，也可能伤害机体。

#### 它主要和什么内容有关？
它和药物作用、治疗作用、不良反应、用药安全这些内容直接相关，也和后面很多合理用药判断连在一起。

#### 它通常在什么阶段出现或被使用？
它通常出现在学习药物基本作用规律的早期阶段，也是进入临床用药分析之前必须先建立的底层认识。

#### 它在整套书和知识体系里放在哪里？
它位于药学专业知识一第二篇药理与毒理学、第四章药物对机体的作用、第一节，是后续理解药效、毒性和用药安全的重要起点。
```

## edges_out — 关系边

节点指向其他节点的边，用于表达知识关系。

| 值 | 含义 | 方向 | 示例 |
|---|---|---|---|
| `isa` | 是个体/实例（具体→抽象） | A → B（A是B的一种） | 美托洛尔 → β受体阻滞剂 |
| `part_of` | 属于整体 | A → B（A是B的一部分） | 抗风湿药 → 解热镇痛抗炎药章节 |
| `mechanism` | 作用机制 | A → B（A通过B发挥作用） | 美托洛尔 → β受体阻滞机制 |
| `causes` | 引起/导致 | A → B（A导致B） | 病原体 → 感染性疾病 |
| `treats` | 治疗 | A → B（A药治疗B病） | 阿司匹林 → 发热 |
| `has` | 含有/具有 | A → B（A含B成分） | 制剂 → 辅料 |
| `relates` | 一般关联 | A → B（A与B相关） | 同领域两个药 |

**边的视觉原则**：
- 不需要为所有"同类"节点都连 `relates` 边——同类关系由 `type` + `category` 的视觉一致性承载即可。
- 边只连真正有知识关联的节点；关联越多边越多，反而视觉噪音越大。
- `reason` 字段用于 hover 时显示边的含义，不需要在图中永久显示所有边标签。
